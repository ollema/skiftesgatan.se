# Time Blocks live in both a TS constant and a DB table

The closed **Time Block** set — currently `[7-10, 10-13, 13-16, 16-19, 19-22]` for the **Laundry Room** and `[7-22]` for the **Outdoor Area** — is reified twice: as a `TIME_BLOCKS` constant in `src/lib/types/bookings.ts` (the current schedule) and as rows in the `time_block` table (every Time Block that has ever existed). `booking.time_block_id` FKs into the table. On the surface this looks redundant — CONTEXT.md calls the set closed, so why isn't the table collapsed into a CHECK constraint or replaced by the constant? It isn't because the schedule has changed in the past and will change again, and we want each historic **Booking** to keep pointing at the exact `(startHour, endHour)` it was made under, regardless of what the current schedule says.

## Context

CONTEXT.md treats Time Blocks as a closed set: "changing or adding a Time Block is a real domain change, not a config tweak." Most reads of a **Booking** today join `time_block` purely to recover the hours, which made the table look like a glorified enum. The `TIME_BLOCKS` constant was introduced to consolidate that closed-set knowledge in one place (driving log prose, the seed scripts, and any future client-side grid enumeration), at which point the table genuinely *looks* superfluous.

The trap: if "closed set" is read as "immutable," the table can be dropped and `booking.start_hour: integer + CHECK` does the same job with one fewer join. But "closed at a point in time" is not "immutable across all time." The schedule has been re-shaped before. When it changes, **Bookings made under the previous schedule still exist** — they appear in iCal feeds, admin views, and reminder emails — and they must keep displaying as the hours they were actually made for, not as whatever the *current* schedule maps to.

## Considered options

- **Drop the `time_block` table; store `start_hour` directly on `booking` with a CHECK constraint.** Cleanest schema, removes 4–5 joins-for-no-reason. Rejected: when the schedule changes, recovering what hours a historic Booking actually ran would require knowing the schedule that was in effect on its `created_at` date — a per-row historical lookup the `time_block` table makes free.
- **Drop the table; keep a versioned `SCHEDULE_HISTORY` constant in TS, keyed by date range.** Same as above but with the historical lookup in code. Rejected: shifts the historical-truth burden into TS, where it'd quietly drift if anyone ever forgot to add a new range. The DB FK is the more honest mechanism — you can't accidentally lose history by editing a constant.
- **Keep the table; add `TIME_BLOCKS` as the source of truth for the *current* schedule (chosen).** The table becomes the historical record; the constant is what code reasons about for live decisions (current grid, log prose for the active schedule, seed). A drift test asserts that `time_block` rows include every entry in `TIME_BLOCKS`, catching the case where a developer edits one without the other.

## Consequences

- **The cache that translates `time_block_id → (resource, startHour, endHour)` must read every row in `time_block`, not just rows matching the current `TIME_BLOCKS`.** Historic Bookings reference Time Block rows whose `(resource, startHour)` is no longer in the current constant; their reminder emails, iCal events, and admin views must still resolve. The cache is the historical-lookup path; `TIME_BLOCKS` is the current-schedule path. Code that resolves hours for an existing Booking must go through the cache. Code that asks "what blocks exist right now?" reads `TIME_BLOCKS`.
- **The drift test asserts inclusion, not equality.** It checks that every entry in `TIME_BLOCKS` is present as a `time_block` row, so the seed and the constant can't disagree about the *current* schedule. It does not assert the table has *only* current-schedule rows — historic rows are expected and load-bearing.
- **Schedule changes are a two-step migration, not an in-place edit.** Add new `time_block` rows for the new blocks, update `TIME_BLOCKS`, and re-point only future Bookings (or just let new Bookings naturally use the new rows). Never delete or mutate an existing `time_block` row that has any Booking referencing it — the FK enforces this, but the discipline matters at the design level too.
- **The seed scripts derive their insert payload from `TIME_BLOCKS` and use `onConflictDoNothing()`** so they're idempotent and additive — running the seed against a database with historic rows leaves them alone.
- **If the schedule ever needed truly atomic re-pointing** (e.g. a bulk "all future Bookings on the old 19–22 block now point at the new 19–20 block"), the table makes that a one-statement migration. The CHECK-constraint alternative would make it a data backfill across the entire `booking` table.
